## Open Issues

### 🔴 CRITICAL #1 — Backend + orchestrator crash-loop on boot → entire site `/api` is 502 (PRODUCTION DOWN)

**Status:** Found 2026-06-08 on deployed `v3.5.9`. Backend pm2 restart counter climbing past **2100** (uptime 0s, never binds :3000); orchestrator restarted **675×**. Frontend (:4200) serves pages, so `GET /` returns 307 → login, but **every `/api/*` call returns 502** (proven externally: `GET /api/user/self → 502`). The app is unusable — no login, no calendar, no posting.

**Error (pm2 backend-error.log, repeating):**
```
Error [ERR_PACKAGE_PATH_NOT_EXPORTED]: No "exports" main defined in
  /app/node_modules/node-telegram-bot-api/package.json
    at exportsNotFound (node:internal/modules/esm/resolve)
    ...
Node.js v22.20.0
```

**Root cause — lockfile/package.json drift:**
- The installed module is `node-telegram-bot-api@1.0.0-rc.0` — a `"type": "module"` ESM-only rewrite whose `package.json` `exports["."]` declares **only** `import` (no `require`/`default`). The backend is compiled to CommonJS and launched with `node --experimental-require-module`; a CJS `require()` of an ESM-only package that has no `require`/`default` export condition throws `ERR_PACKAGE_PATH_NOT_EXPORTED`. The import site is `libraries/nestjs-libraries/src/integrations/social/telegram.provider.ts:13` (`import TelegramBot from 'node-telegram-bot-api'`), loaded at module-init → crashes before Nest boots.
- `package.json:279` correctly declares `"node-telegram-bot-api": "^0.66.0"` (the version the code targets — see comment at `telegram.provider.ts:244` "node-telegram-bot-api ^0.66.0 (Bot API <7)").
- **But `pnpm-lock.yaml` still pins `1.0.0-rc.0`** (importer specifier + resolved version). Docker builds install from the frozen lockfile → the broken RC gets baked into the image.

**How it happened (git):**
- `7c5a34e` (2026-06-07) "chore(deps): security-audit dependency upgrade" bumped the dep `^0.66.0 → 1.0.0-rc.0` in **both** package.json and the lockfile.
- `b0e67f5` "fix(v3.5.1): unblock backend/orchestrator boot" reverted **package.json** back to `^0.66.0` but **did not regenerate the lockfile** → the lockfile drift survived into v3.5.2…v3.5.9. The "v3.5.1 boot-fix" was therefore incomplete.

**Permanent fix (requires rebuild + redeploy):**
1. Regenerate the lockfile to match package.json: `pnpm install --lockfile-only` (resolves `node-telegram-bot-api` back to `0.66.x`). Verify `grep -A2 'node-telegram-bot-api:' pnpm-lock.yaml` shows `0.66.x`, not `1.0.0-rc.0`.
2. Commit lockfile, tag, let CI build `ghcr.io/reaatech/postiz-app:<tag>`, then bump the tag in Coolify and Redeploy.
3. Guard for the future: a CI check that `pnpm install --frozen-lockfile` succeeds (it would have caught the drift), or `pnpm install --lockfile-only && git diff --exit-code pnpm-lock.yaml`.

**Fix in `release/v3.5.10`:** `package.json` reverted to `^0.66.0` *and* lockfile must be regenerated (`pnpm install --lockfile-only`). **Container hotfix:** patched `/app/node_modules/node-telegram-bot-api/package.json` `exports["."]` to add `require`/`default` → `./dist/index.js` (lets the CJS `require` resolve the ESM module; verified `new TelegramBot()` instantiates).

### 🔴 CRITICAL #2 — Backend DI crash: `ProviderHealthService` primitive constructor param

Once #1 was cleared, Nest crashed with `UnknownDependenciesException` on `ProviderHealthService` (dep at index [0], metadata `Object`). Root cause: `provider-health.service.ts` had `constructor(private _defaultThreshold = 5) {}` — a **primitive parameter-property on an `@Injectable()`**. Nest emits `design:paramtypes: [Object]` and tries to DI-resolve it. **Fix:** make `_defaultThreshold` a plain class field; `constructor()` takes no args.

### 🔴 CRITICAL #3 — Backend crash: `IdempotencyFactory` not registered → `startMcp` throws

Next, `startMcp` (`start.mcp.ts:181`) did `app.get(IdempotencyFactory, {strict:false})`, which throws `UnknownElementException` (Nest's proxied `app.get` routes the failure through `ExceptionsZone` → process exit; a `try/catch` does **not** save it) because `IdempotencyFactory` was never added to a module. **Fix:** register `IdempotencyFactory` in `AiModule` providers + exports (it has a no-arg constructor; safe). Also hardened `start.mcp.ts` to resolve it defensively (the in-memory fallback is already guarded everywhere).

### 🔴 CRITICAL #4 — Backend crash: `path-to-regexp` v8 rejects bare `*` wildcard routes

Then `PathError: Missing parameter name at index 8: /agents*`. Nest 11 / Express 5 bundle **path-to-regexp 8.4.2**, which requires *named* wildcards. 5 middleware routes used the old bare `*`: `/agents*`, `/copilot*`, `/ai*` (`api.module.ts`) and `agents/*`, `copilot/*` (`ai.module.ts`). **Fix:** migrate to `{/*splat}` (matches the bare path and everything under it, validated against the installed lib).

### 🔴 CRITICAL #5 — Orchestrator DI crash: `PostsService` needs `RagService`, absent in orchestrator

The orchestrator is a **separate Nest app**; `AiModule` is `@Global` only within the backend. `PostsService` (in `DatabaseModule`) injects `RagService` (index [8], from `AiModule`), which the orchestrator never imported → `UnknownDependenciesException`. **Fix:** orchestrator `app.module.ts` imports `AiModule` **and** `ThrottlerModule.forRoot({storage: ThrottlerStorageRedisService(ioRedis)})` (AiModule's `AiThrottlerGuard` needs ThrottlerModule's tokens), mirroring the backend.

### 🔴 CRITICAL #6 — Orchestrator: Temporal workflow bundle imports disallowed `crypto`

Worker manager then failed: *"Your Workflow code … is importing the following disallowed modules: 'crypto'."* All 6 `post.workflow.v1.0.x.ts` imported `makeId` from `services/make.is` (which does `import crypto from 'crypto'`) to build child workflow IDs (`makeId(10)`). Node `crypto` is banned in the Temporal workflow sandbox (non-deterministic) and the webpack bundler hard-fails. **Fix:** replace `makeId(10)` with replay-safe `uuid4()` from `@temporalio/workflow` in all 6 files; drop the `make.is` import.

**Container restoration:** backend + orchestrator rebuilt from the fixed source (`pnpm run build:backend` / `build:orchestrator`) and run on production dist under pm2 — all three procs online, Temporal worker connected, site serving. (Tried dev/watch mode first; two `nest --watch` processes + the orchestrator's runtime webpack bundling OOM-pressured the host — production dist is the correct lightweight path.)

### Functional issues (from real Playwright UI driving + `e2e/` sweep, post-restore)

**🔴 CRITICAL #7 — Composer can't save/schedule/publish: `POST /api/posts/valid` (+`/preflight`) → 400.**
Driving the real composer UI (open → select channel → type → "Save as draft"): the channel selects fine (button flips "Check the circles above" → "Add to Calendar"), but on submit the composer's pre-checks 400 and the modal hangs open — **no post can be created through the UI.** Captured the exact request the composer sends to `/api/posts/valid`:
```json
{"type":"draft","posts":[{"integration":{"id":"…"},"group":"…",
  "settings":{"who_can_reply_post":"everyone","community":"","firstComment":"",
    "active_thread_finisher":false,"thread_finisher":"…","plug--x-repost-post-users--integrations":[]},
  "value":[{"id":"…","content":"<p>…</p>","delay":0,"image":[]}]}]}
```
The 400 body shows the cause — `/valid` and `/preflight` are typed `@Body() CreatePostDto` but the composer only sends a *partial* body:
- `shortLink`/`date`/`tags` are `@IsDefined` on `CreatePostDto` but absent in the validation payload → *"shortLink/date/tags should not be null or undefined"*.
- Per-post `settings` has **no `__type`**, so the class-transformer discriminated union falls back to base `EmptySettings`, which then rejects every provider field → *"who_can_reply_post / community / firstComment / thread_finisher / plug--… should not exist"* and *"`__type` must be reddit, lemmy, …"*.
Same v3.5.5 regression family (validation-pipe hardening) — the dedicated-lenient-DTO fix existed on `release/v3.5.1` but never reached `main`/v3.5.9. **Fix:** give `/valid` and `/preflight` a dedicated lenient DTO (only `posts` required; `shortLink`/`tags`/`date` optional; per-post `settings` loose via `@Allow()`), since these endpoints only validate content/media — not the full create contract. **Not yet fixed.** (Direct `POST /api/posts` with a full body + `__type` works — that's why the 6am public-API automation still posts; only the composer UI is blocked.)

**🟠 HIGH #8 — Calendar shows no post cards: `GET /api/posts?display=…` → 400 (FIXED live).**
The calendar data fetch (`calendar.context.tsx` `loadData` → `/posts?display=week&customer=&startDate=…&endDate=…`) 400s with `"property display should not exist"`. `GetPostsDto` declared only `startDate`/`endDate`/`customer` — no `display` — so the global `forbidNonWhitelisted` pipe rejects it and **the entire calendar view renders empty** (user-confirmed: "no cards for previous posts"). Same regression class as the v3.5.3 fix, which never reached `main`/v3.5.9. **Fix:** add `@IsOptional() @IsString() display?` to `GetPostsDto`. Verified: `/api/posts?display=week` now 200 returning real posts. Container rebuilt + clean backend restart (had to clear zombie procs holding :3000 first).

**🟡 MEDIUM #9 — CopilotKit `/api/copilot/chat` → 403 on every authenticated page.**
Console error on `/launches`, `/analytics`, `/settings`, etc.: `[CopilotKit] Error (runtime_info_fetch_failed): Runtime info request failed with status 403`. In-app Copilot chat is dead. Known deferred issue from v3.5.3/v3.5.4 — **still present** in v3.5.9 (a CopilotKit runtime request-context nuance; not the AI tier policy). Core AI composer/generation unaffected. Not yet fixed.

**🟡 MEDIUM #10 — `GET /api/integrations/telegram/updates` → 500 (uncaught).**
Was reportedly fixed in v3.5.4 but is 500 again on v3.5.9 (`this.bot.getUpdates(...)`). Likely compounded by the `node-telegram-bot-api@1.0.0-rc.0` API mismatch (the ESM rewrite has a different API than the 0.66.x the code targets). **Re-test after the lockfile→0.66.x permanent fix (#1)** before treating as a separate bug.

**🟢 LOW #11 — `GET /api/user/agent-media-sso` → 400 "Agent Media SSO is not configured".**
Should degrade gracefully (204/empty) instead of a 400 error. Minor; known from prior rounds.

**🟢 LOW/INFO #12 — Global throttle (`90 req/hour`) is easily hit.**
Normal Playwright UI testing tripped the `ThrottlerModule` limit → `429` on `/api/user/self`, `/api/integrations/list` (which breaks the app mid-session). `API_LIMIT` defaults to 90/hr per IP+user; heavy real usage (or a power user with many channels) could hit it. Consider raising the default or scoping it. Counts reset after 1h (Redis TTL).

**Verified working via real UI:** login→`/launches`; calendar Month view renders post cards (#8 fix, screenshot `e2e/calendar-month.png`); channel selection in composer; pages `/launches` `/analytics` `/media` `/campaigns` `/comments` `/settings` render (HTTP 200) when not rate-limited; `find-slot` 200; CRUD create/delete for campaigns. Post-detail modal interaction + remaining pages were inconclusive once the 429 throttle kicked in — re-run after reset.

**Test-harness note:** earlier `e2e/` API-sweep assertions used guessed paths (`/marketplace`, `/settings/team`, `/api/analytics`) — corrected against the real route inventory (`e2e/endpoints.json`). The UI-driving specs (`20–23`) are the authoritative ones.

| **Post detail modal** (v3.3.0) | KPI header, thread, comments | `36-…` shell | assert KPI values, thread expand, comments list/reply | **3** |

**Tier-1 = ship-blocking** (the lifecycle + the invariants that, if broken, mean the app silently
fails for users — exactly the #1–#8 failure mode). **Tier-2 = feature regressions** users would hit
within a session. **Tier-3 = breadth/long-tail.** Note several invariants (SSRF `safeFetch`,
encryption round-trip, RAG, JWT alg-pinning) are **better covered by unit/integration tests in their
packages** than by Playwright — the plan should add those at the `nestjs-libraries` layer, not force
everything through the browser.
